This seed shows a "typical" layout for an application that requires authentication and use roles to distinguish between what logged on users can do.

  • View1 Can be seen by all
  • Users Only Access to data (REST) requires user to be logged on as a user
  • Admins only Access to data (REST) requires user to be logged on as admin

Test Users: (logon with these credentials to test)

  • Admin: username = teacher, password = test
  • User: username= student, password = test

Using the Security Features in this seed - Server:

All REST services placed in the file REST_Admin_API.js requires admin rights

All REST services placed in the file REST_User_API.js requires user rights

Username, password, and roles are hardcoded in index.js (your task is to place this info "somewhere")

Using the Security Features in this seed - Client:

Logon/logof is ready to use.

A token with credentials is automatically is attached to all outgoing AJAX requests (when you are logged on)

In this version users can see all menu items (even if they do not have the necessary rights). This is for you to observe when and where security takes place (bring up F12 in chrome and monitor the network traffic)

You can hide elements so they are only visible with the required security role. Add ng-show="isUser" to the "Users Only" anchor tag in index.html and ng-show="isAdmin" on the "Admins Only" tag. To see this in action.

Everything you do, that relates to security on the client, can be tampered with, so the trick above should be seen as something to make the system more user friendly. The real security is how REST (our data) is protected on the server.

Note: This seed is not meant for production. The most obvious flaw is that username and password are sent unencrypted
